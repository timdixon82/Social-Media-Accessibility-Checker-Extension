## 2. WCAG 2.2 Level A Findings

### A-1. 1.1.1 Non-text Content — badge icons use Unicode symbols without text alternatives

**Severity:** Moderate

In `app.js` the `BADGE_ICONS` map (line 231) prefixes badges with the characters `✓ `, `✗ `, and `! `. These characters are rendered as visible text inside `<span>` elements. Screen readers announce them inconsistently: VoiceOver reads `✓` as "check mark", JAWS may read it as "tick" or skip it, and `✗` is widely announced as "ballot x" rather than "fail". The badge's text label ("Pass", "Fail", "Flag") follows the symbol, so the symbol is redundant for meaning — but because it is literal text content inside the span, not an image, it cannot be suppressed with `aria-hidden`. The labels alone are sufficient; the prefix symbols add noise and inconsistency.

**Recommendation:** Remove the Unicode prefix characters from `BADGE_ICONS`. The label text ("Pass", "Fail", "Flag", "N/A") is sufficient. If a visual symbol is wanted, use an inline SVG with `aria-hidden="true"`.

---

### A-2. 1.3.1 Info and Relationships — duplicate `aria-labelledby` targets across dynamically generated sections

**Severity:** High

`buildAltSection` sets `aria-labelledby="alt-heading"` and `buildEmojiSection` sets `aria-labelledby="emoji-heading"` and `buildFontSection` sets `aria-labelledby="font-heading"` — but these IDs are hardcoded strings (lines 513, 549, 571). When multiple post cards are rendered, each card renders its own instance of these sections. The result is duplicate IDs on the page, which breaks the `aria-labelledby` association: a screen reader will resolve to only the first element with each ID. Cards two onwards have incorrectly labelled sections.

**Recommendation:** Suffix each ID with the per-card sequence counter `seq` already used for `detailsId` and `contrast-h-${seq}`. For example: `aria-labelledby="alt-heading-${seq}"` with `id="alt-heading-${seq}"`.

---

### A-3. 2.1.1 Keyboard — new post card receives programmatic focus unexpectedly

**Severity:** Moderate

`renderPost` calls `card.focus({ preventScroll: true })` (line 445) after appending a card. The card element has `tabindex="-1"` which means it accepts programmatic focus but is not in the tab order. When an audit returns multiple posts rapidly, focus jumps to each new card as it arrives, pulling a keyboard or screen reader user away from any position they had reached earlier in the list.

**Recommendation:** Focus the newly appended card only if the card is the first one (when the list was previously empty) or only after the audit is fully complete. Alternatively, use a polite `aria-live` region to announce new cards without stealing focus.

---

### A-4. 4.1.2 Name, Role, Value — "View full report" button has no accessible name distinguishing it per post

**Severity:** Moderate

Each post card renders a "View full report" button with the same text. With no additional context (no `aria-label`, no `aria-describedby` linking to the post heading), a screen reader listing all buttons on the page presents multiple identical "View full report" items. A JAWS or VoiceOver user cannot tell which post each button refers to.

**Recommendation:** Add `aria-label` such as `"View full report for ${post.author}, ${displayDate}"` to each `viewBtn`.

---

### A-5. 4.1.2 Name, Role, Value — "Expand" / "Collapse" button shares the same accessible name problem

**Severity:** Moderate

The same issue applies to the "Expand" button. All expand buttons have identical text. The `aria-controls` attribute points to the correct `detailsId`, but the button label itself does not identify the post.

**Recommendation:** Add `aria-label` such as `"Expand post by ${post.author}, ${displayDate}"` and update it to "Collapse…" when toggled.

---

### A-6. 1.3.1 Info and Relationships — inline report (`buildReportHtml`) has no `<main>` landmark

**Severity:** Low

The blob-URL report page opened by "View full report" has no landmark structure: no `<main>`, no `<nav>`, no `<header>`. Screen reader users cannot use landmark navigation to jump between sections; they must navigate heading by heading.

**Recommendation:** Wrap the report body content in a `<main>` element. Add a `<header>` for the title/meta block.

## 3. WCAG 2.2 Level AA Findings

### AA-1. 1.4.3 / 1.4.6 Contrast — popup muted text and disabled items

**Severity:** Moderate (code-inspection estimate; requires browser measurement to confirm)

In `popup.html` the `.coming-soon` span uses `color: #999` (line 52) on a `#fff` background. The contrast ratio of `#999` on `#fff` is approximately 2.85:1, which fails both the AA (4.5:1) and AAA (7:1) thresholds for normal text at 11 px. The disabled radio label text (`color: #999`) has the same issue.

The `.url-display` element uses `color: #555` (line 60) on `#f4f4f4`. That ratio is approximately 4.6:1, which passes AA but fails AAA (7:1).

**Recommendation:** Replace `#999` with a colour that achieves at least 4.5:1 on white (for AA) or 7:1 (for AAA). WCAG 2.2 notes that disabled controls are exempt from contrast requirements (SC 1.4.3 exception), but the "coming soon" label beside disabled items is informational text and is not exempt.

---

### AA-2. 4.1.3 Status Messages — progress section uses two simultaneous `aria-live` regions with duplicate content

**Severity:** Low

`app.html` has both `#status-bar` (`aria-live="polite"`) and `#progress-wrap` (`aria-live="polite"`). During an audit both regions update simultaneously: `setStatus` updates the status bar and `showProgress` updates the progress label. A screen reader may announce both in quick succession, which doubles the verbal interruption during long audits.

**Recommendation:** Consolidate status and progress text into a single `aria-live` region, or ensure only one region updates at a time for any given event.

---

### AA-3. 1.4.10 Reflow — popup fixed width

**Severity:** Low

`popup.html` fixes the body width to `300px` (line 11). Chrome extension popups are inherently constrained, but at 400% zoom (beyond 200%, which is the AA threshold) the fixed width causes content to be cut off without a scroll mechanism. This is a structural constraint of the extension popup rather than a page-level authoring choice.

**Recommendation:** Note this as an extension-platform constraint and document it as a project exception. Use `min-width: 300px; max-width: 100%` to allow the popup to flex when the browser zoom is very high.

## 4. WCAG 2.2 Level AAA Gaps

### AAA-1. 1.4.6 Contrast Enhanced — popup body text and UI elements

The popup uses 14 px body text in `color: #1a1a1a` on `#fff`. That ratio is approximately 18.1:1, which passes AAA. However, the `legend` and `field-label` elements use 13 px text, and the `.url-display` uses 11 px text in `#555` on `#f4f4f4` (approximately 4.6:1). For text below 18 px (non-bold) the AAA threshold is 7:1. The url display fails AAA.

**Most pressing AAA gap:** The `#555` on `#f4f4f4` combination used for the current-page URL display in the popup (contrast approximately 4.6:1) fails the AAA 7:1 threshold for 11 px text. This is the most pressing because it displays the current-page URL — functional information a user needs to confirm they are on the correct LinkedIn page before running an audit.

### AAA-2. 2.4.12 Focus Not Obscured (Enhanced) — report page sticky-less but no scroll padding

`app.html` has no sticky header and no scroll-padding rule. Focus obscuration is unlikely in practice, but `scroll-padding-top` is absent. Low risk; worth adding as a one-line CSS addition.

### AAA-3. 2.4.13 Focus Appearance — popup run button focus ring

`button#run-btn:focus-visible` uses `outline: 3px solid #0a66c2` with a 2 px offset. The AAA criterion requires the focus indicator area to be at least as large as a 2 CSS pixel perimeter around the unfocused component, and the contrast of the indicator against adjacent colours must be at least 3:1. The `#0a66c2` focus ring on the `#aaa` (disabled state) background is approximately 3.4:1 — marginal. In the enabled state, the ring appears on the `#0a66c2` button background — a ring of the same colour as the button surface, which produces 1:1 contrast and fails the criterion entirely.

**Recommendation:** Change the focus ring colour for the run button to white (`#ffffff`) or to the dark navy (`#061528`) to ensure it contrasts against both the enabled and disabled button backgrounds.

### AAA-4. 3.1.5 Reading Level — inline report page section headings

The inline report (`buildReportHtml`) uses headings such as "1. Emoji usage", "2. Non-standard Unicode fonts", "3. Image alt text", "4. Image colour contrast". The headings are clear. The body text ("Unicode mathematical character(s) used as decorative text") is technical but short. Reading level is broadly appropriate. No gap.

### AAA-5. 2.2.3 No Timing — no timing issues found

The extension imposes no session timeout. No gap.

### AAA-6. 1.4.8 Visual Presentation — report page line length

`app.html` constrains the page to `max-width: 1100px` with full-width text containers. At wide viewports, text lines in the post-text box may exceed 80 characters. This is a minor AAA gap.

**Recommendation:** Add `max-width: 75ch` or `max-width: 80ch` to the `.post-text` and paragraph elements in the report page.

## 5. Recommended Deferred Items (Later Accessibility Phase)

The following items are real but not blocking for the setup build. They should be tracked in the project wiki and addressed in the first dedicated accessibility sprint.

1. **Manual screen reader testing** on VoiceOver/Safari, JAWS/Chrome, and NVDA/Firefox. Code inspection cannot confirm announcement behaviour for the dynamic card list, the progress region, or the expand/collapse controls. This is required before any public release.
2. **axe-core integration in CI.** The extension's built HTML can be loaded via `file://` in a Playwright harness and scanned with axe-core. This is achievable and should be added to the test workflow.
3. **Keyboard focus management for the card list.** Currently focus jumps to each new card as it is appended (finding A-3). A coherent focus-management strategy — for example, a "Jump to results" link that appears when the first card arrives — should be designed and tested with real screen readers.
4. **`aria-live` region audit with a real screen reader.** The interplay of `#status-bar`, `#progress-wrap`, and the dynamically built card list warrants a dedicated live-region test session to confirm announcements are timely and not duplicated.
5. **Inline report page accessibility.** The blob-URL report opened by "View full report" is a separate HTML document that inherits none of the parent's ARIA attributes. It needs its own landmark structure, skip link, and keyboard focus start point. A full audit of that surface should be done when the report format is stable.
6. **Touch / pointer target size verification.** The expand and view buttons declare `min-height: 44px` in CSS, which meets 2.5.5 (AAA). The popup range slider and radio buttons should be measured on a real device to confirm 44 px minimum target size.
